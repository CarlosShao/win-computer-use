这是一个非常前沿且实用的技术方向。在沙盒模式（Docker + 虚拟桌面）下开发 Agent 的 Skill（技能），核心逻辑是：**将沙盒封装成一个“提供视觉和动作 API 的黑盒”，Agent 在外部通过调用这些 API 来完成任务。**

以下是业界主流的**基于 Docker + Xvfb（虚拟帧缓冲）+ FastAPI 的沙盒 Skill 开发方案**。

------

### 一、 整体架构设计

我们将整个系统分为两部分：

1. **沙盒环境 (Docker 容器)**：运行一个轻量级的 Linux 虚拟桌面，并暴露 HTTP API 供外部调用（截图、点击、输入）。
2. **Agent 大脑 (宿主机或云端)**：运行 LLM 逻辑，通过调用沙盒的 API 来执行 Skill。

```text
[ Agent 大脑 (Python/LangChain) ]
        │  (Function Calling / Tool Use)
        ▼
[ 沙盒控制客户端 (FastAPI Client) ]
        │  (HTTP Requests)
        ▼
[ Docker 沙盒容器 (Ubuntu + Xvfb + VNC + FastAPI Server) ]
   ├── 虚拟屏幕 (Xvfb :99)
   ├── 浏览器 (Chromium)
   └── 动作执行器 (xdotool / pyautogui)
```

------

### 二、 第一步：构建沙盒环境 (Dockerfile)

我们需要一个包含虚拟桌面和 API 服务的 Docker 镜像。创建一个 `Dockerfile`：

```dockerfile
# 使用轻量级 Ubuntu 基础镜像
FROM ubuntu:22.04

# 避免交互式安装提示
ENV DEBIAN_FRONTEND=noninteractive

# 1. 安装虚拟桌面 (Xvfb)、窗口管理器 (fluxbox)、浏览器和控制工具
RUN apt-get update && apt-get install -y \
    xvfb fluxbox x11vnc xdotool scrot \
    chromium-browser \
    python3 python3-pip \
    && rm -rf /var/lib/apt/lists/*

# 2. 安装 Python 依赖 (FastAPI 用于暴露接口)
RUN pip3 install fastapi uvicorn pydantic python-multipart

# 3. 复制我们的 API 服务代码 (见下一步)
COPY sandbox_server.py /app/sandbox_server.py

# 4. 启动脚本：启动虚拟屏幕 -> 启动窗口管理器 -> 启动 API 服务
COPY entrypoint.sh /app/entrypoint.sh
RUN chmod +x /app/entrypoint.sh

EXPOSE 8000
CMD ["/app/entrypoint.sh"]
```

**`entrypoint.sh` 启动脚本：**

```bash
#!/bin/bash
# 启动 1280x800 分辨率的虚拟屏幕
Xvfb :99 -screen 0 1280x800x24 &
export DISPLAY=:99

# 启动轻量级窗口管理器
fluxbox &

# 启动 FastAPI 服务
uvicorn /app/sandbox_server.py:app --host 0.0.0.0 --port 8000
```

### 三、 第二步：开发沙盒底层 API (`sandbox_server.py`)

这个文件运行在容器内，提供 Agent 需要的**基础原子能力**。

```python
import subprocess
import base64
import os
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Coordinate(BaseModel):
    x: int
    y: int

class TextData(BaseModel):
    text: str

# 1. 截图 Skill (视觉感知)
@app.get("/screenshot")
async def get_screenshot():
    # 使用 scrot 截图并保存
    subprocess.run(["scrot", "/tmp/screen.png"])
    with open("/tmp/screen.png", "rb") as f:
        # 返回 base64 编码的图片，方便直接传给多模态 LLM
        return {"image_base64": base64.b64encode(f.read()).decode("utf-8")}

# 2. 鼠标点击 Skill (动作执行)
@app.post("/click")
async def click_mouse(coord: Coordinate):
    # 使用 xdotool 移动并点击
    subprocess.run(["xdotool", "mousemove", str(coord.x), str(coord.y)])
    subprocess.run(["xdotool", "click", "1"]) # 1 代表左键
    return {"status": "clicked", "x": coord.x, "y": coord.y}

# 3. 键盘输入 Skill (动作执行)
@app.post("/type")
async def type_text(data: TextData):
    # xdotool type 对中文支持不好，这里演示英文。中文通常用 xclip + 粘贴
    subprocess.run(["xdotool", "type", "--clearmodifiers", data.text])
    return {"status": "typed", "text": data.text}

# 4. 执行命令 Skill (系统级操作)
@app.post("/bash")
async def run_bash(cmd: TextData):
    result = subprocess.run(cmd.text, shell=True, capture_output=True, text=True)
    return {"stdout": result.stdout, "stderr": result.stderr}
```

### 四、 第三步：开发 Agent 侧的高级 Skill (Tool)

在 Agent 大脑侧，我们将上述原子 API 封装成 LLM 能理解的**高级 Skill**，并使用**装饰器模式**解决你关心的“执行时间过长”问题。

#### 💡 核心技巧：如何处理耗时操作？（异步截图对比法）

如果 Agent 点击了“下载”按钮，需要等 10 秒。不能让 LLM 傻等，也不能让 LLM 盲目乱点。
**最佳实践**：执行动作后，**定时轮询截图，对比屏幕差异，直到屏幕稳定或出现预期元素**。

**Agent 侧代码示例 (Python)：**

```python
import requests
import base64
import time
import cv2 # 用于简单的图像差异对比
import numpy as np

SANDBOX_URL = "http://localhost:8000"

class SandboxController:
    def __init__(self):
        self.last_screenshot = None

    def get_screenshot(self):
        res = requests.get(f"{SANDBOX_URL}/screenshot").json()
        self.last_screenshot = res["image_base64"]
        return res["image_base64"]

    def click(self, x, y):
        requests.post(f"{SANDBOX_URL}/click", json={"x": x, "y": y})

    def type_text(self, text):
        requests.post(f"{SANDBOX_URL}/type", json={"text": text})

    # 🌟 高级 Skill：智能等待屏幕变化 (解决耗时问题)
    def wait_for_screen_change(self, timeout=15, threshold=0.05):
        """
        等待屏幕发生显著变化（如页面加载完成、弹窗出现）
        """
        start_time = time.time()
        # 获取初始屏幕
        initial_img = self._decode_image(self.get_screenshot())
        
        while time.time() - start_time < timeout:
            time.sleep(1) # 每秒检查一次
            current_img = self._decode_image(self.get_screenshot())
            
            # 计算两张图片的差异度 (简单的像素差值)
            diff = cv2.absdiff(initial_img, current_img)
            non_zero_count = np.count_nonzero(diff)
            total_pixels = diff.size
            change_ratio = non_zero_count / total_pixels
            
            # 如果变化率超过阈值，认为屏幕已更新
            if change_ratio > threshold:
                print(f"检测到屏幕变化 ({change_ratio:.2%})，继续执行...")
                return self.last_screenshot
                
        print("等待超时，屏幕无显著变化。")
        return self.last_screenshot

    def _decode_image(self, b64_str):
        # 将 base64 转为 OpenCV 格式用于对比
        img_bytes = base64.b64decode(b64_str)
        nparr = np.frombuffer(img_bytes, np.uint8)
        return cv2.imdecode(nparr, cv2.IMREAD_COLOR)

# 实例化控制器
ctrl = SandboxController()

# ==========================================
# 定义给 LLM 调用的 Tool (Skill)
# ==========================================

def skill_open_and_search(keyword: str) -> str:
    """
    Agent 调用的高级 Skill：打开浏览器并搜索关键词
    """
    # 1. 打开浏览器
    ctrl.bash(f"chromium-browser --start-maximized https://google.com &")
    
    # 2. 智能等待页面加载完成 (核心防坑机制)
    ctrl.wait_for_screen_change(timeout=10)
    
    # 3. 此时 LLM 应该介入，分析截图并给出搜索框的坐标 (x, y)
    # 这里为了演示，假设我们已经知道搜索框在 (640, 400)
    search_box_x, search_box_y = 640, 400 
    
    # 4. 执行点击和输入
    ctrl.click(search_box_x, search_box_y)
    ctrl.type_text(keyword)
    
    # 5. 按下回车
    ctrl.bash("xdotool key Return")
    
    # 6. 再次等待搜索结果加载
    ctrl.wait_for_screen_change(timeout=10)
    
    # 7. 返回最终截图给 LLM 进行下一步分析
    return ctrl.get_screenshot()
```

### 五、 总结：这套方案的优势与避坑指南

#### 1. 为什么这样设计？

- **解耦**：LLM 不需要知道底层是 `xdotool` 还是 `pyautogui`，它只需要调用 `click(x,y)`。
- **安全**：所有操作都在 Docker 内，即使 Agent 执行了 `rm -rf /`，也只会销毁容器。
- **可观测性**：你可以随时通过 VNC 连入容器（端口 5900），用肉眼观察 Agent 在虚拟桌面里到底在干嘛。

#### 2. 开发避坑指南（血泪经验）

- **分辨率陷阱**：LLM 对坐标的预测高度依赖截图分辨率。**务必在 Docker 中将 Xvfb 的分辨率固定**（如 1280x800），并且在 Prompt 中明确告诉 LLM：“屏幕分辨率是 1280x800，请基于此给出坐标”。
- **输入法问题**：`xdotool type` 无法直接输入中文。**解决方案**：在沙盒内安装 `xclip`，将中文文本写入剪贴板，然后模拟 `Ctrl+V` 粘贴。
- **时间感知**：像上面代码中的 `wait_for_screen_change` 极其重要。如果没有它，Agent 点击“提交”后，页面还没跳转，它就立刻截图分析，会导致严重的“幻觉”和操作错位。

这套方案是目前开发 Computer Use Agent 最标准、最稳健的工程化路线，你可以直接基于此框架扩展出“自动填写表单”、“自动抓取数据”等各种复杂的 Skill。
