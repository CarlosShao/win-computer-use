## 二、关键修正点（执行时必须遵守）

以下是在评审过程中发现的坑点，明天写代码时必须注意：

1. **RapidOCR 不是"零配置"** —— 首次运行仍需自动下载模型 ~40MB（中文+英文），网络不好一样卡住。文档措辞应改为："无需系统级配置，首次运行自动下载模型（可预下载缓存）"
2. **`smart_click()` 不要引入新库** —— 复用已有的 `pywinauto`（即 `ui_find.py`），不要引入 `uiautomation` 库。pywinauto 的 UIA 后端已经够用。
3. **OCR 不全面替换** —— 接入 RapidOCR 作为默认/推荐后端，但保留 Tesseract 作为可选后端（用户可通过参数切换）
4. **多尺度只做 3 个比例** —— `[1.0, 1.25, 1.5]` 对应 Windows 常见 DPI 缩放（100%/125%/150%），不需要更多
5. **"只需改十几行代码"略有夸大** —— 实际 OCR 后端切换改动约 5-10 行核心代码 + 错误处理，整体可控

## 五、讨论记录

### 2026-06-26：Hermes Computer Use 的"专属指针"观察

**现象**：
用户使用 Hermes 的 computer use 功能时，发现 AI 有自己的鼠标指针和键盘控制，不抢用户的输入设备。AI 可以打开浏览器等操作，用户同时也可以点击干扰到它。

**分析**：

这不是真正的"完全隔离"，而是**输入层面的部分隔离**：

| 隔离类型 | 含义 | Hermes 做到了吗？ |
|---------|------|------------------|
| 输入隔离 | AI 的鼠标/键盘事件不和用户冲突 | ✅ 看起来有独立指针 |
| 视觉/交互隔离 | 用户不能点击/干扰 AI 打开的窗口 | ❌ 没有（用户仍能点它的窗口）|

**可能的技术实现**（按可能性排序）：
1. **后台鼠标事件（SendInput 到指定窗口）** —— 最可能，成本低
2. **Windows 虚拟桌面（CreateDesktop API）** —— 每个桌面有独立输入队列
3. **RDP 远程桌面会话** —— 每个会话有独立光标
4. **虚拟机 / Windows Sandbox** —— 完全隔离但开销大
5. **虚拟 HID 驱动** —— 创建第二个鼠标设备（需驱动签名，极高门槛）

**结论**：
- Hermes 大概率用的是**方案 1（后台发送鼠标事件）**，不是真正的虚拟设备
- 对于 `win-computer-use`：短期可用 `pywinauto.click_input()` 实现类似效果；长期如需真正隔离，用 Windows Sandbox

---

### 2026-06-26："成本高"的真实含义

**问题**：用户多次看到"高成本"标注，不确定是指资金还是其他。

**澄清**：

| 标注 | 实际含义 | 具体工作量 |
|------|---------|-----------|
| **低成本** | 改几个现有文件，几小时搞定 | 半天以内 |
| **中成本** | 新功能模块，需要设计 + 编码 + 调试 | 1-2 天 |
| **高成本** | 架构级变更或涉及算法攻关 | 3 天 ~ 1 周+ |

**与资金无关** —— 所有工具都是免费的（Python、GitHub、Windows），这里的"成本"纯粹指**人力开发时间**和**技术复杂度**。

具体例子：
- `smart_click()` 高成本 ≈ 改 4-5 个文件 + ~300 行新代码 + 半天调试边界情况
- MCP Server 高成本 ≈ 全新模块 ~1000 行 + 学习 MCP 协议 + 架构决策
- `--with-markers` 高成本 ≈ 计算机视觉算法研究 + 实现 + 可能一周以上

## 六（补）、市面产品实测结论（2026-06-27 下午）

### 测试对象与结果

| 产品 | 实测现象 | 是否有沙箱/真隔离 | 用户能否干预 |
|------|-----------|-------------------|-------------|
| **Hermes Computer Use** | 有独立光标视觉反馈，但操作的是真实屏幕，用户可干扰 | ❌ 无 | ✅ 能 |
| **Codex（Windows）** | 桌面频繁闪烁，顶部显示"Codex is using your computer"，看不到输入过程，但可点击其他部分干扰 | ❌ 无 | ✅ 能 |
| **Claude Code** | 仅支持 Mac，无法在 Windows 测试 | — | — |
| **Open Code** | 无 computer use 功能 | — | — |

### 核心结论

> **没有任何一款市面产品能做到真正的沙箱独立操作、防止手动干预。**

所有产品的"隔离"都是**假象**：
- Hermes 的"独立光标" = 视觉反馈，底层还是操作真实屏幕
- Codex 的提示条 = 告知用户"我在操作"，但没有任何拦截机制
- 用户随时可以抢鼠标、删内容、点别的地方

### 对本项目的影响

**✅ 不需要做沙箱**——连大厂产品都没做出来，说明这条路要么技术上极难，要么产品上没必要。

**✅ 采用"诚实方案"**：
- 操作前通知用户（弹出提示条："Agent 正在操作，请勿干扰"）
- 操作过程中显示进度
- 操作完成后通知
- 如果真的需要防干扰：可选加键盘鼠标锁（带超时自动释放安全机制）

**❌ 不需要研究 Hermes 多光标方案**——实测证明它并不是真正的多光标，只是视觉上的，研究价值不大。
